# 技术实现细节

## 1. 实现边界

当前增强主要发生在飞书插件源码中，不在 OpenClaw core 中重写协作运行时。

边界是：

- 插件内负责群消息识别、模式判定、协作上下文注入、handoff 状态机
- 插件内不负责 durable shared workspace 真相源

这是刻意的 KISS 选择。

## 2. 关键源码落点

### 2.1 配置兼容

- `src/config-schema.ts`
- 继续沿用官方 `channels.feishu` 配置 schema

### 2.2 群协作模式判定

- `src/mention.ts`
- 负责 `default / direct_reply / peer_collab / coordinate` 的群模式分类

### 2.3 原始群消息仲裁

- `src/monitor.account.ts`
- 负责哪个账号应该响应、哪个账号应该跳过

### 2.4 runtime 协作注入

- `src/bot.ts`
- 负责把协作上下文注入到 agent 当前轮运行时

### 2.5 协作状态机

- `src/collaboration.ts`
- 负责：
  - `task_id`
  - owner
  - `speakerToken`
  - `agent_handoff`
  - `accept/reject/need_info/complete`
  - 状态惰性回收

### 2.6 出站可见层清洗

- `src/reply-dispatcher.ts`
- 负责：
  - 流式 coalesce
  - 剥离内部控制块
  - 避免把协作控制 JSON 发到群里

### 2.7 群参与者视图

- `src/chat.ts`
- `src/chat-schema.ts`
- 负责 `feishu_chat(action="participants")`

## 3. 当前工程判断

### 已经做的

- 群协作主模式
- owner / handoff 状态机
- 基本资源回收
- 流式重复回复收敛
- 内部控制块泄漏修复
- 工具账号路由修复

### 还没做的

- durable shared workspace 真相源
- 完整 task store
- 完整 session 生命周期治理
- 外部 bot 参与内部协作

## 4. 为什么不继续强化原生 bot-to-bot `@`

因为飞书群里的原生 bot-to-bot `@` 不是可靠控制面。

所以现在的原则是：

- visible `@` 保留，用来维持群里的协作感
- 真正的接棒和多轮协作，用内部协议完成

这比继续把展示层 `@` 硬做成控制层更稳。

## 5. 开发与同步

README 首页现在只保留“怎么使用这个插件”。

如果需要继续开发这份源码仓库，最小流程如下：

```bash
npm install
npm test
npm run sync:local
systemctl --user restart openclaw-gateway.service
```

`npm run sync:local` 会同步到当前机器上实际生效的 OpenClaw Feishu 插件安装目录。这个路径依赖你的 Node/OpenClaw 安装方式，不应在通用文档里写死。
