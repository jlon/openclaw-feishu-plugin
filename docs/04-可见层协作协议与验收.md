# 可见层协作协议与验收

这份文档只定义一件事：群里看起来应该如何协作。

控制层已经由插件内部的 `direct_reply / peer_collab / coordinate + owner + handoff` 负责；
这份文档约束的是用户在群里真正能看到的行为，避免出现“内部协议是对的，但群里看起来像假讨论”的情况。

生产主路径默认使用自然群聊：
- 用户直接点多个内部 bot
- `main` 在后台做入口分类
- runtime 控制后续轮次和可见 `@`

显式模式 `#直答 / #协作 / #编排` 仍然存在，但这里只作为 override/debug，不再是唯一主路径。

## 1. 核心原则

1. 可见层只服务于群体验，不承担真实控制职责。
2. 一次可见回复只说当前 bot 自己这一层，不替别人作答。
3. 轻量共同点名不是讨论，不要制造假交接。
4. 复杂协作必须看得出谁先说、谁接棒、谁收口。
5. 不允许出现内部控制块、工具调用、会话路由、调试语句。

## 2. 三种可见层模式

### 2.1 `direct_reply`

适用场景：
- 同时点名多个 bot
- 问的是轻量问题
- 不要求持续讨论或汇总

可见规则：
1. 每个被点名 bot 只回复自己的一句。
2. 不 cue 其他 bot。
3. 不说“轮到谁”“我把话头交给谁”“收到”“协作完成”。
4. 不追加第二条确认消息。

正确示例：
- `稳。`
- `先看 Flink 任务状态和最近异常。`

错误示例：
- `我先说，接下来交给 Starrocks-SRE。`
- `我说完了，等对方回复。`
- `两位都已回答完成。`

### 2.2 `peer_collab`

适用场景：
- 同时点名多个内部专业 bot
- 明确要求一起看、互相补充、继续讨论、协作定位

可见规则：
1. 初始阶段每个参与者只说一句自己的第一判断。
2. owner 进入下一轮时，先补自己的一层，再明确交棒给下一个参与者。
3. target 接续时，直接从自己的角色继续补一层，不显式说“收到接力棒”“轮到我”“我来接手”。
4. 每一轮说完立刻停，不追加“收到”“等待对方”“协作完成”。
5. 最后一轮由当前 owner 用一句自然话收口。

正确节奏：
1. `Flink-SRE：我先从实时计算链路看延迟是 source、state 还是 sink。`
2. `Starrocks-SRE：从查询和导入侧看，更像是结果层先暴露出来。`
3. `Flink-SRE：目前更像 sink 变慢放大了延迟，先收这个判断。`

错误节奏：
1. `我把话头交给对方。`
2. `收到接力棒，我来接手。`
3. `两位都已完成。`
4. `sessions_send(...)`

### 2.3 `coordinate`

适用场景：
- 明确要求 `main` 安排、分工、汇总

可见规则：
1. 原始群消息只由 `main` 接。
2. `main` 可见回复分三段：
   - 接单
   - 阶段性进展
   - 汇总结论
3. 专业 bot 可以在群里发可见补充，但不抢原始入口，不替 `main` 汇总。
4. `main` 不需要反复 @ 所有人，只在有必要时明确点名。

## 3. 10 个真实群验收场景

### 场景 1：单 bot 轻量直答

输入：
```text
@Flink-SRE 一个字描述下 john
```

通过标准：
1. 只有 `Flink-SRE` 回复。
2. 回复只有一条。
3. 回复不带其他 bot 的 @。

不通过表现：
1. `main` 抢答。
2. 回复带 `@Starrocks-SRE`。
3. 同一 bot 连发两条。

### 场景 2：两个 bot 轻量共同点名

输入：
```text
@Flink-SRE @Starrocks-SRE 你俩各用一句话说下你们先看什么
```

通过标准：
1. 两个 bot 各回一句。
2. 不出现交棒话术。
3. 不出现“完成/收到/等对方”。

### 场景 3：三个 bot 轻量共同点名

输入：
```text
@Flink-SRE @Starrocks-SRE @SoulCoder 你们各用一句话介绍自己
```

通过标准：
1. 三个 bot 各回自己一句。
2. 不互相 cue，不汇总。

### 场景 4：两个 bot 进入 `peer_collab`

输入：
```text
@Flink-SRE @Starrocks-SRE 你俩协作讨论什么是灵魂，先各自判断，再互相补充，最后形成一句结论
```

通过标准：
1. 首轮能看到两个 bot 的第一判断。
2. 后续至少有一次明确接棒。
3. 最终只有一句自然结论收口。

不通过表现：
1. 只有各答一句，没有后续接棒。
2. 全程只有 `main` 或只有一个 bot。
3. 出现内部 JSON 或工具调用。

### 场景 5：`@main + 多专业 bot` 的编排

输入：
```text
@首席大管家 @Flink-SRE @Starrocks-SRE 帮我安排并汇总这次排查
```

通过标准：
1. 原始群消息由 `main` 接。
2. `main` 明确说明安排谁看什么。
3. 最终由 `main` 汇总。

### 场景 6：外部 bot + `main`

输入：
```text
@云上Bot @首席大管家 你俩各用一句话赞美下我
```

通过标准：
1. `main` 只说自己的那一句。
2. 不把外部 bot 串成内部 agent。
3. 不出现错误 @ 到内部 bot。

### 场景 7：bot 问 bot，但不进入 fake mention-forward

输入：
```text
@Flink-SRE @Starrocks-SRE 你俩互相打个招呼
```

通过标准：
1. 两个 bot 各自说自己的话。
2. 不出现 `@对方` 的重复前缀污染。
3. 不出现“我已经通知对方”。

### 场景 8：达到 hop 上限后自然收口

前置：`maxHops=1`

输入：
```text
@Flink-SRE @Starrocks-SRE 你俩继续下钻讨论这条链路，最后形成一句结论
```

通过标准：
1. 初始接棒仍可发生。
2. 达到上限后，当前 owner 自己收口。
3. 不把结论甩回用户，也不要求第三个 bot 接手。

### 场景 9：不泄漏内部控制块

输入：
```text
@SoulCoder @Starrocks-SRE #协作 你俩讨论下什么是灵魂
```

通过标准：
1. 群里不出现 `collab_assess`、`agent_handoff`、JSON、fence。
2. 只看到正常可见文本。

### 场景 10：流式输出不重复关卡

输入：
```text
@首席大管家 #编排 你来收集下大家的爱好
```

通过标准：
1. 群里只看到一次最终可见回复。
2. 不出现同一轮多次 close streaming 的重复回复。
3. 不因为缺权限重复报相同错误。

## 4. 验收顺序

建议按这个顺序验：
1. 场景 1、2、3：先确认 `direct_reply` 干净。
2. 场景 4、8、9：再确认 `peer_collab` 像真人一样接棒且不泄漏。
3. 场景 5、10：最后确认 `main` 编排和流式输出。
4. 场景 6、7：补外部 bot 和 bot 间共同点名边界。

## 5. 当前判断标准

只要出现下面任一条，就不算通过：
1. 出现内部 JSON、fence、工具调用。
2. 轻量直答出现假交棒或假完成。
3. 复杂协作看不出接棒过程。
4. `main` 和专业 bot 抢答同一入口。
5. 外部 bot 被串成内部 agent。
