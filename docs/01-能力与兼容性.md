# 能力与兼容性

说明：文中的 `main` 只是示例协调账号 id。实际运行时的协调账号取自 `channels.feishu.defaultAccount`，不要求固定叫 `main`。

## 1. 目标

这份文档只回答两件事：

1. 这个插件相对官方 stock 飞书插件多了什么
2. 当前是否兼容原生 OpenClaw 飞书插件的参数能力

## 2. 当前保留的原生能力

当前仍然保留并兼容官方飞书插件的基础能力：

- 多账号配置
- `defaultAccount`
- `dmPolicy`
- `groupPolicy`
- `allowFrom`
- `groupAllowFrom`
- `requireMention`
- `renderMode`
- `streaming`
- `blockStreamingCoalesce`
- `groupSessionScope`
- `replyInThread`
- `tools.doc/wiki/drive/perm/scopes/chat`

结论：**参数兼容层面，当前是增量增强，不是破坏性替换。**

## 3. 当前新增的能力

在官方 stock 插件之上，当前增强版主要补的是“群协作协议层”，不是重写飞书基础接入。


### 3.1 群协作协议增强

新增了 4 种群协作模式：

- `default`
- `direct_reply`
- `peer_collab`
- `coordinate`

并保留了可选的显式模式标签：

- `#直答`
- `#协作`
- `#编排`

这些标签不是新的 schema 字段，而是群消息里的协议头。它们现在是 override/debug 能力，不再是日常主交互。

### 3.2 结构化协作状态

新增了：

- `task_id`
- owner
- `speakerToken`
- `agent_handoff`
- `accept/reject/need_info/complete`
- `collaboration.maxHops`

### 3.3 工具增强

新增了：

- `feishu_chat(action="participants")`

它回答的是：

- Feishu 成员接口可见的人类成员
- 当前 OpenClaw 内部、配置上可参与这个群的内部机器人

它不是飞书官方原生能力，而是插件补出来的参与方视图。

## 4. 真实痛点与当前解法

### 痛点 1：每次只能稳定点一个 Agent

增强后：
- 支持共同点名多个内部 Agent
- 用 `direct_reply / peer_collab / coordinate` 区分不同场景
- 日常主路径仍然是自然群聊
- `#直答 / #协作 / #编排` 只保留成 override/debug

### 痛点 2：群里越聊越乱，互相乱 `@`

增强后：
- 可见 `@` 继续保留在展示层
- 真正的控制面改到内部协作协议
- 共同点名直答和复杂协作严格分流

### 痛点 3：A 想让 B 接手，但 B 接不到

增强后：
- 不再依赖飞书原生 bot-to-bot `@` 做控制面
- 改为 `owner + handoff + accept/reject/need_info/complete`
- 并增加 `maxHops` 限制持续下钻深度，避免失控和 token 浪费

## 5. 显式模式与兼容层

当前主路径已经改成：
1. 用户自然点多个内部 bot
2. 协调账号在后台做一次入口分类
3. runtime 接管 task、轮次、停止条件和可见 `@`

显式模式保留，但只在需要强制覆盖时使用：

```text
@A @B #直答 ...
@A @B #协作 ...
@协调账号 @A @B #编排 ...
```

这层设计的目的不是让用户学 DSL，而是给调试、回归和强制模式覆盖留一个稳定入口。

## 6. 不兼容判断

当前没有证据表明我们破坏了官方配置 schema。

但需要明确两点：

1. 官方没有群多 Agent 协作协议
2. 官方没有把飞书群里的 bot-to-bot `@` 当成可靠控制面

所以如果有人把当前能力理解成“官方 stock 插件已经原生支持群里多个 bot 多轮协作”，这个理解是错的。

## 7. 当前结论

- **配置参数层面：兼容官方 stock 插件**
- **运行时协作能力层面：是显式增强版，不是 stock 行为**
- **发布时应明确标识为增强版飞书插件，不要伪装成 upstream 原样行为**
