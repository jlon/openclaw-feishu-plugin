# 群聊协作模式

## 1. 模式总览

当前插件支持 4 种群聊协作模式。

生产主路径是自然群聊。

用户只需要正常点多个内部 bot，插件会先让 `main` 在后台做一次入口分类，然后由 runtime 接管协作推进。

显式模式：
- `#直答`
- `#协作`
- `#编排`

现在只作为 override/debug 能力；没有显式模式时，系统会先按自然消息做稳定分类。

### `default`

适用场景：
- 无 `@`
- 普通群任务入口

行为：
- 只有 `main` 处理

### `direct_reply`

适用场景：
- 同时点名多个 Agent，但只是轻量直答
- 例如：打招呼、一个字描述、各自介绍

行为：
- 每个被点名 Agent 只代表自己说一句
- 不创建协作任务
- 不进入 handoff

常见写法：

```text
@A @B 你俩各说一句 ...
```

### `peer_collab`

适用场景：
- 同时点名多个内部专业 Agent
- 明确要求持续协作、多轮讨论、受控下钻

行为：
- 创建协作任务
- 初始并行判断
- 选出当前 owner
- 允许结构化 handoff
- 支持受控多轮协作
- 支持持续下钻，但默认最多成功接棒 `3` 次

常见写法：

```text
@A @B 你俩一起看下这条链路，先各自说判断，再互相补充
```

### `coordinate`

适用场景：
- 明确要求 `main` 安排、拉通、汇总

行为：
- 只有 `main` 处理原始群消息
- 专业 Agent 通过内部协作参与
- 最终由 `main` 汇总

常见写法：

```text
@首席大管家 @A @B 帮我安排并汇总
```

## 2. 自然群聊主路径与显式 override

当前推荐的交互方式是：
1. 日常使用自然群聊
2. 插件在后台做分类
3. runtime 推进协作

显式模式仍然保留，因为有些场景需要强制覆盖系统判断。

### 2.1 自然群聊主路径

自然多 bot 点名时，当前分类规则是：
1. 命中轻量直答模式 -> `direct_reply`
2. 命中明确编排模式 -> `coordinate`
3. 其他多内部 bot 点名 -> 默认 `peer_collab`

这里的关键约束是：
- `main` 是隐藏入口分类器
- `main` 不默认进入 `peer_collab` 的参与者集合
- 真正的轮次、可见 `@` 和停止条件都由 runtime 决定

### 2.2 显式模式为什么还保留

原因不是“写标签更酷”，而是：
1. 自然语言是开放集
2. 模式判定是控制协议
3. 用开放集去驱动控制协议，本身就会不稳

所以当前插件保留两条路径：
1. 自然群聊：主路径
2. 显式模式：override/debug

## 3. 关键原则

1. 可见 `@` 是展示层
2. 稳定控制面是内部协作协议
3. 外部 bot 不进入内部 handoff 主链
4. 共同点名直答和复杂协作必须严格区分

## 4. 典型例子

### 轻量共同点名直答

```text
@Flink-SRE @Starrocks-SRE 一个字描述下 john
```

预期：
- 两个 Agent 各回自己的一句
- 不创建协作任务

### 专业 Agent 协作

```text
@Flink-SRE @Starrocks-SRE 你俩一起看下这条链路，先各自说判断，再互相补充
```

预期：
- 进入 `peer_collab`
- 允许多轮协作

## 5. 持续下钻与 `maxHops`

当前插件支持“持续下钻”，但它不是无限互相转发。

控制方式：

```json
{
  "channels": {
    "feishu": {
      "collaboration": {
        "maxHops": 3
      }
    }
  }
}
```

也可以按账号覆盖：

```json
{
  "channels": {
    "feishu": {
      "accounts": {
        "main": {
          "collaboration": {
            "maxHops": 2
          }
        }
      }
    }
  }
}
```

规则：
1. `maxHops` 只限制结构化 `handoff`
2. 初始并行判断不计入 `maxHops`
3. 只有成功 `accept` 的接棒才计数
4. 达到上限后，当前 owner 仍可继续回答或完成当前阶段，但不能再继续 handoff

建议：
1. 日常群排障先用默认值 `3`
2. 如果你只想让两个 Agent 做一次短协作，再收口，用 `1`
3. 不要一开始就开很大，先观察 owner 和 handoff 行为是否稳定

### main 编排

```text
@首席大管家 @Flink-SRE @Starrocks-SRE 帮我安排并汇总这次排查
```

预期：
- 进入 `coordinate`
- 只有 `main` 接原始群消息
- 由 `main` 编排并最终汇总

## 6. 自然语言兼容层

当前自然语言分类不再走“保守直答”策略，而是：

1. 明显轻量句式 -> `direct_reply`
2. 明确编排/安排/汇总 -> `coordinate`
3. 其余多内部 bot 共同点名 -> `peer_collab`

显式模式仍然存在，但只负责强制覆盖：
- `#直答`
- `#协作`
- `#编排`
