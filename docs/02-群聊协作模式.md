# 群聊协作模式

## 1. 模式总览

当前插件支持 4 种群聊协作模式。

生产主路径推荐直接写显式模式：

- `#直答`
- `#协作`
- `#编排`

没有显式模式时，才会进入兼容性的自然语言分类。兼容层默认保守，不再把“辩论/讨论/互相补充”直接升级成 `peer_collab`。

### `default`

适用场景：
- 无 `@`
- 普通群任务入口

行为：
- 只有 `main` 处理

### `direct_reply`

适用场景：
- 同时点名多个 Agent，但只是轻量直答
- 例如：打招呼、一个字描述、各自介绍

行为：
- 每个被点名 Agent 只代表自己说一句
- 不创建协作任务
- 不进入 handoff

推荐写法：

```text
@main #直答(A,B) ...
```

### `peer_collab`

适用场景：
- 同时点名多个内部专业 Agent
- 明确要求持续协作、多轮讨论、受控下钻

行为：
- 创建协作任务
- 初始并行判断
- 选出当前 owner
- 允许结构化 handoff
- 支持受控多轮协作
- 支持持续下钻，但默认最多成功接棒 `3` 次

推荐写法：

```text
@main #协作(A,B) ...
```

### `coordinate`

适用场景：
- 明确要求 `main` 安排、拉通、汇总

行为：
- 只有 `main` 处理原始群消息
- 专业 Agent 通过内部协作参与
- 最终由 `main` 汇总

推荐写法：

```text
@main #编排(A,B) ...
```

## 2. 显式模式为什么更稳

原因不是“写标签更酷”，而是：
1. 自然语言是开放集
2. 模式判定是控制协议
3. 用开放集去驱动控制协议，本身就会不稳

所以当前插件保留两条路径：
1. 显式模式：生产主路径
2. 自然语言分类：兼容层

如果你要稳定发布给别人用，应该优先教用户写 `#直答 / #协作 / #编排`。

## 3. 关键原则

1. 可见 `@` 是展示层
2. 稳定控制面是内部协作协议
3. 外部 bot 不进入内部 handoff 主链
4. 共同点名直答和复杂协作必须严格区分

## 4. 典型例子

### 轻量共同点名直答

```text
@首席大管家 #直答(Flink-SRE,Starrocks-SRE) 一个字描述下 john
```

预期：
- 两个 Agent 各回自己的一句
- 不创建协作任务

### 专业 Agent 协作

```text
@首席大管家 #协作(Flink-SRE,Starrocks-SRE) 你俩一起看下这条链路，先各自说判断，再互相补充
```

预期：
- 进入 `peer_collab`
- 允许多轮协作

## 5. 持续下钻与 `maxHops`

当前插件支持“持续下钻”，但它不是无限互相转发。

控制方式：

```json
{
  "channels": {
    "feishu": {
      "collaboration": {
        "maxHops": 3
      }
    }
  }
}
```

也可以按账号覆盖：

```json
{
  "channels": {
    "feishu": {
      "accounts": {
        "main": {
          "collaboration": {
            "maxHops": 2
          }
        }
      }
    }
  }
}
```

规则：
1. `maxHops` 只限制结构化 `handoff`
2. 初始并行判断不计入 `maxHops`
3. 只有成功 `accept` 的接棒才计数
4. 达到上限后，当前 owner 仍可继续回答或完成当前阶段，但不能再继续 handoff

建议：
1. 日常群排障先用默认值 `3`
2. 如果你只想让两个 Agent 做一次短协作，再收口，用 `1`
3. 不要一开始就开很大，先观察 owner 和 handoff 行为是否稳定

### main 编排

```text
@首席大管家 #编排(Flink-SRE,Starrocks-SRE) 帮我安排并汇总这次排查
```

预期：
- 进入 `coordinate`
- 只有 `main` 接原始群消息
- 由 `main` 编排并最终汇总

## 6. 自然语言兼容层

当前仍保留兼容性分类，但策略已经改成保守模式：

1. `@main + 安排/协调/汇总` 倾向 `coordinate`
2. 明显轻量句式倾向 `direct_reply`
3. 其余多 bot 共同点名默认仍是 `direct_reply`

只有显式 `#协作` 才稳定进入 `peer_collab`。发布文档和用户培训应把显式模式放在第一位。
